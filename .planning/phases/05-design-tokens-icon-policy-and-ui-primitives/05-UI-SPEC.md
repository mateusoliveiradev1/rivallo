# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Intent and Scope

The Phase 5 foundation must make Rivallo feel like a precise football-management command centre before any product screen exists. Its working direction is **Arquitetura tática**, with minimal heraldic influence restricted to future brand expression, and its north star is **“Sala de comando sob os refletores.”** Strength comes from hierarchy, data craft, restrained geometry, legible icons, and confident density.

This contract covers semantic design tokens, generic and football-specific icon policy, accessible shared primitives, a semantic dense-table primitive, and an internal deterministic UI Lab. It does not authorize a dashboard, squad screen, tactical pitch, product navigation, persistence, real football data, final mark, literal mascot, or final brand identity.

## Design System

| Property | Value |
|----------|-------|
| Tool | Custom Rivallo primitives; no shadcn initialization, templates, theme, or generated component code |
| Preset | Not applicable |
| Component library | Native HTML first; narrowly scoped Radix behavior primitives for composite interactions that need managed focus/keyboard behavior |
| Icon library | `lucide-react` behind one Rivallo wrapper, plus an original versioned football SVG registry |
| Operational font | Inter for body, controls, labels, data, forms, tables, and statistics |
| Restricted title font | Space Grotesk is provisional and permitted only for the working mark, level-one headings, and rare special moments |
| Styling identity | Rivallo semantic tokens and component styles only; Radix supplies behavior, never appearance |

The dependency direction is fixed: authored tokens and icon registries → Rivallo primitives → deterministic UI Lab. Primitives and the Lab must not depend on Tauri commands, the local lifecycle service, API contracts, application/domain logic, persistence, or network availability.

## Layout and Density Contract

- Desktop-first, compact density is the default. The smallest target viewport is 1366×768; mobile composition is outside this phase.
- Surfaces form real regions through tone and restrained 1px borders. Avoid cards nested inside cards and avoid using elevation to compensate for weak hierarchy.
- Controls default to 32px height. Touch-target expansion may use surrounding hit area where needed, but must not visually inflate dense desktop rows.
- Default DenseTable rows and headers are 32px high. Its Lab-only comfortable density is 40px. Type never shrinks below 12px.
- Normal panels use 8px radius; compact controls use 6px; menus/popovers may use 8px; dialogs may use 12px. Pills are reserved for status/chip shapes.
- Elevation uses semantic levels: 0 normal surface, 1 raised control, 2 menu/popover, 3 dialog. Tonal separation and borders remain primary; shadows must be narrow, low-opacity, and absent at level 0.
- Layer order is fixed: base (0), sticky (100), dropdown/popover (300), modal backdrop (400), modal (500), toast (600), tooltip (700). Component CSS consumes semantic layer tokens rather than arbitrary values.

## Typography

The canonical token inventory is 12, 14, 16, 18, 20, 24, 30, and 36px. The isolated Typography category in the UI Lab must demonstrate all eight tokens, but that catalog is an inventory—not a valid example of using all eight sizes in one interface.

Every real UI Lab composition outside that isolated catalog has a strict budget of at most four sizes—**12, 14, 18, and 24px**—and at most two weights—**400 and 600**. A specimen does not have to use the entire budget. The 16, 20, 30, and 36px tokens remain canonical and visible in the isolated inventory, but are not used in Phase 5 specimen, navigation, control, table, or evidence compositions. All operational UI uses Inter. Statistics, finance, positions, dates, ranks, and table numbers use `font-variant-numeric: tabular-nums`.

| Token / role | Size | Weight | Line Height | Contract |
|--------------|------|--------|-------------|----------|
| Dense data and metadata | 12px | 400 or 600 | 16px / 1.333 | Operational; Inter only; tabular numerals where applicable |
| Body and default control | 14px | 400 or 600 | 20px / 1.429 | Operational; button labels use 600 |
| Inventory token 16 | 16px | 400 or 600 | 24px / 1.5 | Isolated typography-token catalog only in Phase 5 |
| Section heading | 18px | 600 | 24px / 1.333 | Operational; hierarchy through weight and spacing |
| Inventory token 20 | 20px | 400 or 600 | 26px / 1.3 | Isolated typography-token catalog only in Phase 5 |
| Contextual/page heading | 24px | 600 | 30px / 1.25 | Operational; provisional Space Grotesk is allowed only at this level |
| Inventory token 30 | 30px | 400 or 600 | 36px / 1.2 | Isolated typography-token catalog only in Phase 5 |
| Inventory token 36 | 36px | 400 or 600 | 42px / 1.167 | Isolated typography-token catalog only in Phase 5 |

Long labels must wrap or truncate intentionally with a discoverable full value; they must not overlap controls. Table headings may wrap to two lines only in comfortable density; compact density uses a concise visible heading plus accessible full name where necessary.

## Color

### Composition and roles

| Role | Provisional candidate | Usage |
|------|-----------------------|-------|
| Dominant (60%) | `canvas: oklch(0.145 0.014 190)` | Continuous application and Lab canvas; never pure black |
| Secondary (30%) | `surface: oklch(0.185 0.018 190)` and `surface-raised: oklch(0.225 0.020 190)` | Panels, table regions, navigation, menus, raised controls |
| Accent (≤10% combined) | Emerald `oklch(0.72 0.15 155)`; cyan `oklch(0.78 0.12 220)`; aged gold `oklch(0.75 0.10 85)` | Primary/positive, information/focus, and rare brand/achievement roles respectively |
| Destructive | Red `oklch(0.68 0.18 25)` | Error, injury, suspension, defeat, critical risk, and destructive confirmation only |

Accent is reserved for: one primary action in a region; confirmed-positive status; focus and informational status; attention/error indicators; rare brand or achievement proof; and small contextual club details. It is never a general decoration layer.

The candidates below are **provisional authored OKLCH values**, not approved final tokens. Before approval, the implementation must convert or gamut-map them to the actual browser-targeted sRGB result and record passing contrast for every declared foreground/background pair. A failed pair must be adjusted; aesthetic preference never waives WCAG thresholds.

| Semantic token | Provisional candidate | Required use and limit |
|----------------|-----------------------|------------------------|
| `color-canvas` | `oklch(0.145 0.014 190)` | Dominant base, never `#000` or equivalent pure black |
| `color-surface` | `oklch(0.185 0.018 190)` | Primary work surface |
| `color-surface-raised` | `oklch(0.225 0.020 190)` | Menus, raised controls, active panel regions |
| `color-overlay` | `oklch(0.105 0.010 190 / 0.78)` | Modal backdrop only |
| `color-border` | `oklch(0.330 0.022 190)` | Structural separators and control boundaries |
| `color-border-subtle` | `oklch(0.270 0.018 190)` | Table row dividers and quiet grouping |
| `color-text` | `oklch(0.955 0.008 185)` | Operational text and data; cold near-white |
| `color-text-strong` | `oklch(0.985 0.004 185)` | Exceptional emphasis only, not every heading |
| `color-text-muted` | `oklch(0.760 0.018 185)` | Secondary copy that must still pass normal-text contrast |
| `color-action-primary` | `oklch(0.720 0.150 155)` | Primary action and confirmed positive only |
| `color-on-action-primary` | `oklch(0.165 0.018 175)` | Text/icons on emerald controls |
| `color-info` / `color-focus` | `oklch(0.780 0.120 220)` | Informational status and visible focus ring |

| `color-danger` | `oklch(0.680 0.180 25)` | Critical/error/destructive semantics with icon and text cue |
| `color-premium` | `oklch(0.750 0.100 85)` | Working mark, achievement, and rare premium moment only |
| `color-selection` | `oklch(0.320 0.055 210)` | Selected row/control surface plus border/icon/text cue |

### Explicit color limits

- Neutral canvas and graphite surfaces must occupy at least approximately 90% of a normal proof viewport. The 60/30/10 rule is an upper-bound composition guide, not permission to fill 10% with saturated colour.
- Aged gold must not exceed a small brand/achievement detail (target ≤2% of a normal viewport), must never be a primary action, and must not become table chrome.
- Emerald appears on the single primary action in a region and on confirmed-positive status. Secondary and quiet actions remain neutral.
- Red and amber never style ordinary navigation, neutral data, or decorative borders. Large solid red/amber panels are prohibited; use restrained tinted surfaces plus icon and explicit copy.
- Cyan is informational and the focus colour; it must not compete with emerald for action priority.
- Club colours are limited to crests, kits, flags, and small contextual details. They must not recolour the application shell or carry selection/status alone.
- No gradient is required. Purple-blue gradients, neon bloom, glass overlays, and casino-like gold coverage are prohibited.

### Contrast contract

- Normal text: at least 4.5:1 against every surface on which it is approved.
- Large text: at least 3:1 only when it meets WCAG size/weight definitions; do not use the lower threshold for operational labels.
- Focus rings, control boundaries, selected indicators, and meaningful iconography: at least 3:1 against adjacent colours.
- Disabled controls remain identifiable without suggesting availability; explanatory text must retain readable contrast even when the control surface is subdued.
- Every status combines colour with a visible label and, where useful, an icon or shape. No state, club, nation, selection, injury, warning, or error relies on colour alone.

## Copywriting Contract

Rivallo copy is concise, operational, and calm. Use direct Brazilian Portuguese verbs, sentence case, and specific recovery instructions. Avoid casino, betting, hype, military aggression, or vague “something went wrong” language. Loading, empty, error, and offline text must state what is happening and what the user can do next.

| Element | Copy |
|---------|------|
| Primary CTA | `Aplicar estado` |
| Empty state heading | `Nenhum exemplo disponível para este estado.` |
| Empty state body | `Selecione outro estado no controle acima para continuar a inspeção.` |
| Error state | `Não foi possível renderizar este exemplo. Revise a configuração e tente novamente.` |
| Destructive confirmation | `Confirmar ação`: `Esta demonstração confirma apenas o comportamento do diálogo. Nenhuma ação real será executada.` |
| Destructive cancel action | `Cancelar` |
| Destructive confirm action | `Confirmar ação` |
| Navigation collapse | `Recolher navegação` |
| Navigation expand | `Expandir navegação` |

The destructive wording belongs only to an AlertDialog proof fixture. Phase 5 contains no real destructive product action.

## Icon Contract

- The only generic family is Lucide, consumed through a Rivallo `Icon` boundary. Consumers choose a semantic icon name and approved size; they do not mix packages, arbitrary paths, stroke widths, or visual styles.
- Approved sizes are 16px for compact inline/table use, 20px for controls and navigation, and 24px for section/empty-state emphasis. Default optical stroke is approximately 1.75px.
- Generic icons use `currentColor`. Decorative icons are `aria-hidden`; meaningful standalone icons require an accessible name at the controlling element.
- Icon plus visible label is the default. An icon-only control is allowed only in a stable, familiar location and must have an accessible name and keyboard-accessible tooltip. A tooltip supplements rather than replaces the accessible name.
- If no icon communicates the concept unambiguously, use text until an icon is approved.
- Football-specific icons are original, versioned SVGs in a typed registry. They use coherent 16/20/24 view boxes, geometric/minimal construction, and the same optical weight as Lucide. No copied crests, third-party game paths, emoji, or mixed icon family is permitted.
- Every used generic icon and every football SVG is shown in the UI Lab at 16/20/24px on graphite, with name, meaning, decorative/semantic behavior, tooltip behavior where applicable, and focus/state evidence.
- Icons remain static. Only real loading/progress may animate, and the reduced-motion rendering must be static or instantaneous.

## Primitive Inventory and State Matrix

Every interactive primitive must prove all applicable states: default, hover, focus-visible, active, selected, disabled, loading, and error. The UI Lab state control must render actual component state, not a cosmetic class preview.

| Primitive | Foundation | Required contract |
|-----------|------------|-------------------|
| Button | Native `<button>` | 32px compact height, 6px radius; primary, secondary, quiet, and destructive-proof variants; optional 16px leading icon; loading preserves width and accessible name |
| IconButton | Native `<button>` + Tooltip | 32×32px visible control; 16/20px icon; mandatory accessible label; tooltip in stable icon-only use |
| TextField | Native labelled `<input>` | Visible label above; 32px control; helper or error below; stable IDs and `aria-describedby`; error is text plus icon/border |
| Select | Native `<select>` where sufficient | Same 32px label/help/error geometry; custom composite selection requires justified Radix behavior and equivalent keyboard semantics |
| Checkbox | Native input | Label is the hit target; checked, unchecked, indeterminate, disabled, focus, and error-group proof |
| RadioGroup | Radix behavior or native radios | Arrow-key movement, visible group label, selected cue beyond colour, and error association |
| Switch | Radix behavior | Visible label and explicit state text where ambiguity exists; Space/Enter operation; disabled and focus proof |
| Tabs | Radix behavior | Arrow-key traversal, roving focus, selected text/indicator, and stable panel association |
| Tooltip | Radix behavior | Supplemental description only; opens on keyboard focus/hover, dismisses with Escape, never contains interactive content |
| Popover | Radix behavior | Contextual non-modal work, explicit trigger, outside/Escape close, predictable focus return |
| Menu | Radix behavior | Arrow/Home/End/Escape keyboard path, labelled trigger, disabled items, and checked items where relevant |
| Dialog | Radix behavior | Genuine interruption only; managed initial focus, trapped modal focus, labelled title/description, Escape/close, focus returned to invoker |
| AlertDialog | Radix behavior | Destructive confirmation proof only; `Cancelar` receives safe default emphasis; no real mutation |
| Toast | Radix behavior or equivalent live region | Brief non-critical feedback; dismissible/expiring without hiding important state; persistent conditions remain contextual |
| Status | Semantic status/alert region | Neutral, info, positive, warning, danger, offline/loading examples; icon plus visible label; appropriate polite/assertive live behavior |
| Skeleton | Inert presentation | Preserves expected layout; hidden from accessibility tree where redundant; non-animated reduced-motion form |
| Empty State | Semantic region | Exact approved heading/body and useful next-step control where applicable; no decorative illustration dependency |
| Error State | Alert/status region | Exact approved recovery copy; impact and correction path remain visible; retry action only when meaningful |
| Pagination | Native buttons/nav | Labelled navigation region; current page communicated in text/ARIA, disabled boundaries, compact controls, no fake data fetching |
| ScrollArea | Native overflow first | Keyboard/trackpad scroll remains native; visible overflow affordance; no hidden content or custom scrollbar dependency for meaning |
| DenseTable | Semantic HTML table | Contract detailed below; no div-grid substitute and no production data model |

### Shared state rules

- Hover is enhancement only; keyboard and touch-equivalent operation must remain complete.
- Focus-visible uses a 2px cyan semantic outline with 2px offset, meets 3:1 non-text contrast, is never clipped, and remains distinct from selection.
- Active state gives immediate 100ms feedback through restrained tonal/position change; it does not bounce or scale dramatically.
- Selected state combines surface/border or marker with visible text/icon semantics.
- Disabled controls are native-disabled when possible, cannot receive action, and do not use a loading cursor as a substitute for state.
- Loading controls expose `aria-busy` or an equivalent announced state, retain their label context, prevent duplicate action, and do not cause layout shift.
- Error state associates explanatory text programmatically with the affected control and never relies on a red border alone.

## DenseTable Contract

DenseTable is a reusable semantic primitive and the main proof of Rivallo's information craft. Its Phase 5 fixture may resemble football data only enough to test layout; it is not a squad schema or product screen.

- Markup begins with `<table>`, `<caption>` (visible or accessible as appropriate), `<thead>`, `<tbody>`, `<th scope="col">`, and `<td>`. A labelled scroll region provides horizontal overflow.
- Sticky headers are supported without obscuring focus or losing column alignment. The sticky layer uses the semantic `sticky` layer and an opaque graphite surface.
- Default compact row/header height is 32px with 8px horizontal cell padding. Comfortable Lab density uses 40px height and 12px horizontal padding. Both use 12px or 14px Inter and tabular numerals.
- Numeric values align right; textual identifiers align left; short categorical/status values use consistent alignment. Headers align with their data.
- Sorting uses a real button inside the header and visible direction text/icon. The sorted header exposes `aria-sort`; unsorted headers do not imply an order.
- Row selection uses an explicitly labelled checkbox column. Space toggles the focused checkbox, selected rows expose a non-colour cue, and bulk-selection state is announced.
- Column definitions include deterministic priority. At constrained widths, lower-priority columns are intentionally hidden or reached through horizontal overflow; text is never compressed below the type floor.
- A Lab-local column menu uses labelled checkbox controls. Density and visibility changes live only in component state, reset on reload, and have no storage, preference, reordering, saved-view, or API behavior.
- Each row exposes at most one visible primary action with text where space permits. Secondary actions use a labelled keyboard-accessible overflow menu.
- Loading preserves column geometry with inert skeleton rows. Empty and error states remain inside the labelled table region and present the approved copy/recovery action. No state replaces the table with an unrelated card.
- Nationality fixtures use a small flag plus country code; the full country name appears in a tooltip/detail. The code remains present if the image fails, so nationality is not represented only by flag or colour.
- Pagination proves navigation semantics only. Server sorting, server pagination, virtualization, data fetching, persistence, and real player/club actions are out of scope.

## UI Lab Contract

### Access and isolation

- The Lab uses a dedicated internal development route such as `/__ui-lab`, guarded by `import.meta.env.DEV` at the entry boundary.
- It is absent from production navigation and production route registration/build behavior. A production attempt must resolve to the normal operational shell or a not-found state without shipping a hidden product tool.
- The Lab renders from committed deterministic TypeScript fixtures. It must not invoke Tauri, require the lifecycle sidecar, call HTTP, read persistence, import contracts-client data, or depend on network availability.

### Information architecture

The Lab navigation contains exactly these proof categories:

1. Semantic tokens — semantic name, authored OKLCH value, resolved/gamut-mapped value, approved foreground/background pair, measured contrast ratio, threshold, and real usage example.
2. Typography — complete fixed scale, Inter operational examples, restricted Space Grotesk examples, tabular statistics, long Portuguese copy, and localisation-length stress.
3. Icons — every consumed Lucide icon and original football SVG at 16/20/24px, semantic/decorative behavior, labels/tooltips, and optical comparison.
4. Primitives — the complete inventory and actual applicable state matrix.
5. DenseTable — compact/comfortable density, visibility controls, sort, selection, row actions, nationality, sticky header, and loading/empty/error.
6. Accessibility evidence — keyboard order, visible focus, non-colour status, long text, 200% zoom/text spacing, reduced motion, and contrast results.
7. Shell proof — expanded and collapsed navigation composition only; no product navigation destinations or dashboard content.

Each category has a clear heading and short purpose statement. The primary state-fixture action is `Aplicar estado`.

Every UI Lab proof follows one focal sequence: **contextual heading → dominant specimen canvas → secondary controls and evidence**. The specimen canvas receives the strongest surface contrast and spatial emphasis; controls and evidence support it without becoming competing panels. Only the specimen's primary action may use emerald. Category navigation, viewport/state selectors, density/column controls, and all surrounding Lab chrome remain neutral graphite/near-white, with cyan reserved for focus or informational meaning. This sequence and colour priority must remain visually intact at 1366×768, 1920×1080, and 2560×1080.

### Shell proof

- Expanded navigation is 232px wide and shows 20px icons plus visible labels. Collapsed navigation is 56px wide and shows the same icons with accessible names and keyboard tooltips.
- Collapse is triggered explicitly by a 32×32px control labelled `Recolher navegação`; the collapsed control is labelled `Expandir navegação`.
- The workspace expands when navigation collapses without changing content order or losing focus. Focus remains on the toggle after the transition.
- State is local and resets on reload. Persistence and real AppShell ownership are deferred.

### Viewport presets and responsive behavior

| Preset | Required proof |
|--------|----------------|
| 1366×768 | Compact density remains usable; expanded/collapsed navigation, all primary controls, table header/actions, focus, and horizontal overflow are visible without text shrinking |
| 1920×1080 | Default desktop composition shows stable hierarchy and useful density without unnecessary empty panels |
| 2560×1080 | Ultrawide composition preserves grouping; documentation content is bounded and dense surfaces use width intentionally rather than stretching labels/paragraphs |

- Presets constrain a labelled preview frame and show its exact dimensions. They are deterministic evidence, not a claim of browser/device emulation.
- At 1366px, low-priority DenseTable columns hide by declared priority or remain available through a clearly labelled horizontal scroll region. Controls may wrap by group; labels do not disappear.
- At 1920px, the Lab uses its standard navigation and content proportions.
- At 2560px, prose/specimen columns cap at 1680px; full-width table specimens may extend to 2200px when doing so reveals useful columns. Empty width must not be filled by giant cards or oversized metrics.
- At 200% zoom/effective constrained width, the Lab may reflow into one column and use controlled overflow. No action, label, dialog, or focus target may become unreachable.

## Motion Contract

| Token | Duration | Usage |
|-------|----------|-------|
| `motion-feedback` | 100ms | Press, hover, focus feedback |
| `motion-control` | 150ms | Menu, tooltip, switch, selection transitions |
| `motion-layout` | 200ms | Navigation collapse and local layout adjustment |
| `motion-emphasis-max` | 250ms | Rare state transition; requires functional purpose |

- Use ease-out; no bounce, elastic spring, parallax, page-load choreography, neon glow, or decorative continuous motion.
- Navigation collapse animates only width/opacity where performance and focus remain stable. Content must not sweep across the screen.
- Loading indicators represent real waiting. Skeleton animation, if present, is low-contrast and must not become a visual focal point.
- Under `prefers-reduced-motion: reduce`, nonessential transitions and skeleton/icon animation become instantaneous or static. No workflow may depend on animation to communicate change.

## Accessibility and Interaction Evidence

The baseline is WCAG 2.2 AA, not a later remediation target.

- Complete keyboard path: reach category navigation, state controls, every primitive, DenseTable sort/selection/overflow, shell toggle, overlays, and dismissal without a pointer.
- Logical focus order follows reading order. Modal focus enters the dialog, remains contained, and returns to the invoker. Popovers/menus/tooltips close with Escape according to their pattern.
- Every form control has a visible label. Helper/error text is programmatically associated. Placeholder text is never the only label.
- Status updates use appropriate `role="status"`, `role="alert"`, `aria-live`, and `aria-busy` behavior without repetitive announcements.
- Semantic headings and table relationships survive styling. Do not use clickable `<div>` elements or ARIA to recreate native controls.
- Long text fixtures include long player/club-like names, two-line labels, and at least 200% expansion. Content may wrap or expose a discoverable full value; it may not overlap or be clipped without access.
- Reduced motion has visible Lab evidence and automated browser coverage.
- Automated accessibility scans are supporting evidence, not a substitute for manual keyboard, focus, screen-reader naming, contrast, and optical review.

## Test and Review Evidence

Approval requires all of the following:

1. Token integrity test proves required semantic roles/scales, rejects pure-black canvas, and reports the failing pair and measured ratio.
2. Deterministic generated-token drift check regenerates or compares without modifying tracked files and prints an actionable repair command.
3. Component tests assert accessible roles/names, label/help/error relationships, focus behavior, disabled/loading semantics, non-colour state cues, and composite keyboard behavior.
4. DenseTable tests assert native table semantics, `aria-sort`, labelled selection, local density/visibility controls, and loading/empty/error structure.
5. A production-boundary test proves the development-only Lab is unavailable in production behavior and independent of Tauri/API/service state.
6. Playwright evidence covers 1366×768, 1920×1080, and 2560×1080; keyboard traversal; focus visibility; reduced motion; long text; and deterministic screenshots/visual comparison where configured.
7. Manual review confirms original football SVG optical balance at 16/20/24px and the premium hierarchy at all three viewports.
8. `pnpm quality`, the non-watch UI Lab browser suite, and desktop build pass from a previously clean checkout; generated caches/artifacts are ignored and `git status --porcelain` remains clean.
