# Phase 06.1: SM-2 Table View Engine and Durable Views - Context

**Gathered:** 2026-07-16
**Status:** Ready for planning
**Mode:** Delegated autonomous decisions; the user explicitly asked the agent to continue through SM-6 without pausing between phases

<domain>
## Phase Boundary

Implement the canonical Table View Engine as a reusable controlled capability and migrate the real Elenco table to it. This phase owns the current fixed-club screen contract, pure view-state commands, a versioned `local-fixed` repository behind Tauri/application ports, legacy preference migration, the complete saved-view lifecycle, and accessible table customization.

This phase does not implement selectable career identity, SQLite, server queries, cache/offline synchronization, dashboard data, multiplayer sharing, grouping UI, or a second product table. Those remain Phase 7/9 or later integration work. Existing AppShell, routes, squad semantics, inspector, XI, Tactics and match simulation must remain intact.

</domain>

<decisions>
## Implementation Decisions

### Controlled state and Elenco schema

- Use `05-TABLE-VIEW-ENGINE-CONTRACT.md` as the sole behavioral contract; do not create a competing table-state model.
- The first owning-screen contract uses stable `tableId: squad.primary`, `schemaVersion: 1`, `ownerScope: local-fixed`, stable semantic `columnId`s, stable player `rowId`s, required identity/action columns and explicit capability flags.
- Required columns are shirt number, lineup information, player identity and position. They remain visible and explain why they cannot be hidden; labels, translated copy and render order never become identity.
- Implement controlled visibility, order, finite width, start/end pinning, density, sort and supported filters. Grouping remains declared unsupported and therefore has no active control.
- Use the bounded current-club result with a declared client-pagination window large enough for the 18-player fixture; do not introduce virtualization or server-query claims before measured Phase 9 data.

### Saved views, ownership and durability

- Preserve the current default position ordering and all existing filters while routing their durable form through engine commands. Quick UI filters may remain transient only when the screen contract explicitly marks them transient.
- Support system-default, user-owned and read-only provenance plus activate, create, duplicate, rename, delete, set-default, reset, save, save/discard/cancel and textual dirty-state behavior.
- `Mostrar somente gols` must be proven through the ordinary lifecycle as a user-owned view: required identity columns plus goals remain visible, goals sort descending and a typed `goals > 0` filter is applied. No `goalsOnly` flag or renderer/query branch is allowed.
- Rust application/platform code owns an atomic app-data repository (separate from `first-playable.json`) with versioned envelopes, sequential migrations, quarantine reason/evidence and deterministic fallback. React components never call `localStorage`, files or Tauri persistence directly.
- A narrow frontend migration adapter may read `rivallo.squad-ui.v2` through `v4` once, normalize supported density/columns and submit an import command. It removes or retires legacy keys only after confirmed repository persistence; invalid/unknown-only payloads recover textually to the valid system default.
- `local-fixed` is an explicit temporary owner seam, not a multi-user claim. Phase 9 must be able to migrate it to career identity/SQLite without changing view semantics.

### Customizer interaction and visual preservation

- Keep Elenco as the workspace and preserve its top hierarchy, table, minifaces, real shirt numbers, GOL-to-ATA ordering, inspector and current AppShell. Improve the table controls incrementally; do not redesign the entire page.
- Use a saved-view selector plus one controlled, portalled, non-modal customization surface anchored to the existing table controls. It may become a responsive sheet-like surface at 1024px, but must not replace the table or create nested cards.
- The customizer provides column search, required-state explanation, pointer reorder, keyboard move controls, visibility, finite-width adjustment, pin/unpin, immediate preview, explicit save/cancel, partial reset and full reset. Unsupported options are absent.
- Header resize/reorder has pointer and keyboard parity. Escape cancels the active operation, accepted changes keep focus on the affected column, and polite announcements name the column plus resulting position/width/pinning.
- Preserve current row density geometry unless measured code evidence requires a compatible token adjustment. At widths at least 1280px the table keeps at least 65% of the workspace with the inspector open; at 1024px the inspector reflows without removing table/actions or creating document overflow.
- Continue the Rivallo dark teal-graphite system, Inter operational type, tabular numerals, 12px minimum operational text, radius at most 12px, tokenized focus/contrast and 100–250ms state motion with an instant reduced-motion path.

### Validation, dependencies and non-regression

- Install no new registry dependency. The Tauri/Rust serde boundary remains authoritative and a small typed TypeScript decoder guards frontend input; adding Zod without the project's explicit package-approval process is out of scope for this safe slice.
- Preserve selection by player ID when sorting/filtering/reordering columns. Pinned/sticky columns must never cover focused content, and the table keeps semantic native table relationships.
- Treat loading, empty, invalid, repository unavailable, corrupted/migrated/recovered and save-failure states as real product states with text plus non-colour cues.
- Tests must cover pure commands/normalization/dirty comparison, schema rejection, repository round-trip, atomic replacement, legacy v2-v4 migration, sequential migration, future-schema/corruption quarantine, saved-view lifecycle, pointer/keyboard customization, focus restoration, restart persistence and `Mostrar somente gols` through the general pipeline.
- Full existing component, browser matchday, Rust, architecture, formatting, typecheck, lint and Tauri build gates must remain green before Phase 06.2 begins.

### the agent's Discretion

- Exact module/file names, finite width bounds, maximum pinned-column count, localized copy and internal command/event representation may follow existing conventions as long as the canonical contract, scope boundary and measurable UI/a11y constraints above remain true.
- Prefer the smallest reusable pure-state engine and existing native/Radix primitives. Do not introduce state/query/router libraries merely to make the implementation look more modern.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `apps/desktop/src/matchday/MatchdayScreen.tsx` currently owns `rivallo.squad-ui.v4`, legacy v2/v3 normalization, filters, sort and preference mutations; it is the migration source but must stop being storage authority.
- `apps/desktop/src/matchday/SquadWorkspace.tsx` already preserves a semantic native table, controlled density/column popovers, stable player selection, tooltips, minifaces and the right inspector. It should consume the engine rather than be replaced.
- `apps/desktop/src/matchday/matchday-ui.ts` contains stable existing column/sort labels and the current density/optional-column vocabulary that can seed the Elenco schema.
- `crates/platform/src/matchday.rs` demonstrates an injected application repository with atomic temporary-file replacement and app-data location composition; the table-view repository should follow the pattern without coupling to matchday state.
- `apps/desktop/src-tauri/src/main.rs` already composes the matchday coordinator and Tauri commands at the host boundary.
- Existing Radix Popover/Tooltip primitives, semantic z-index tokens, Status/Toast feedback and Playwright/component harnesses are reusable.

### Established Patterns

- React owns presentation and transient interaction; Rust application/platform boundaries own persistence and competitive authority.
- File-backed local state is injected through a repository, serialized deterministically, written through a temporary file and composed from the Tauri app-data directory.
- Table rows are native `<table>` relationships and player selection is keyed by stable IDs.
- No unreviewed dependency transaction is permitted; the current desktop intentionally has no Router, Query, Zustand or Zod package.

### Integration Points

- Replace `UiPreferences.density` and `visibleColumns` ownership with a table-view controller while leaving sidebar, active screen, inspector visibility and pitch mode in the existing UI-preference seam.
- Extend the Tauri client/host with load/save/import table-view commands backed by application/platform services.
- Keep current squad sort/filter functions as screen-owned football semantics invoked from normalized view descriptors.
- Update component and browser fixtures at 1024×768, 1366×768, 1920×1080 and 2560×1080 without changing the normal production entry route.

</code_context>

<specifics>
## Specific Ideas

- The user wants a professional dense manager table, not another UI Lab, and explicitly asked for the screen to remain usable while capabilities improve incrementally.
- Visual references confirm Elenco and Tactics remain separate screens; the table stays the main workspace and the right inspector remains contextual.
- Existing numbers, faces, nationality flags and football semantics must remain visible and truthful. No rating, grouping or data-depth filler may be invented to populate the customizer.

</specifics>
