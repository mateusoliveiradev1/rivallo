# Phase 06.1 Research — Table View Engine and Durable Views

<user_constraints>

## User Constraints (copied from CONTEXT.md)

- Implement the canonical Table View Engine in real Elenco; use `05-TABLE-VIEW-ENGINE-CONTRACT.md` as the sole behavior model.
- Fix the first owner to `tableId: squad.primary`, `schemaVersion: 1`, `ownerScope: local-fixed`; preserve AppShell, routes, Elenco, inspector, XI, Táticas and gameplay.
- Required columns are shirt number, lineup information, player identity and position; they remain visible and explain why.
- Support visibility, order, finite width, start/end pinning, density, sort and supported filters; grouping is unsupported and has no control.
- Keep bounded client pagination with a window large enough for the 18-player fixture.
- Preserve default position ordering and existing filters; only explicitly transient quick filters may remain outside durable state.
- Support system-default, user-owned/read-only provenance and the complete activate/create/duplicate/rename/delete/default/reset/save/dirty-transition lifecycle.
- Prove `Mostrar somente gols` as a normal user-owned view with required columns, goals visible, goals descending and typed `goals > 0`.
- Do not add `goalsOnly` or any renderer/query special case.
- Rust application/platform owns a separate atomic app-data repository, versioned envelopes, migrations, quarantine and fallback; React never owns durable storage.
- A narrow frontend adapter may read v2–v4 once; retire legacy keys only after supported density/columns are confirmed durable.
- Invalid or unknown-only legacy payloads recover textually to the valid system default.
- Keep `local-fixed` visibly temporary so Phase 9 can migrate identity/storage without changing table semantics.
- Use one portalled non-modal customizer with search, explanations, pointer/keyboard parity, visibility, width, pinning, preview, save/discard and reset.
- Escape cancels the active operation; accepted changes retain focus and announce column plus resulting position/width/pinning.
- Preserve current density geometry unless measured evidence requires a compatible token change.
- At ≥1280px the table retains 65% with inspector open; at 1024px the inspector reflows without hiding table/actions or overflowing the document.
- Keep dark teal-graphite, Inter, tabular numerals, 12px minimum text, radius at most 12px and reduced-motion parity.
- Install no dependency; preserve native table semantics, player-ID selection and unobscured focus.
- Treat loading, empty, invalid, unavailable, corrupt, migrated, recovered and save-failure states as real textual states.
- Validate commands, normalization, dirty state, schemas, durability, migration, quarantine, lifecycle, keyboard/pointer parity, restart and goals pipeline.
- Defer identity, SQLite, large-scale query modes, cache/offline/sync, sharing, grouping and other tables to Phase 9 or later.

## Summary

Phase 06.1 should be implemented as two coordinated but separately testable systems:

1. a pure controlled TypeScript engine and Elenco schema that normalize commands and render proposals;
2. a Rust application port plus platform file repository that validates and durably owns `local-fixed` saved views.

The screen controller loads once, then emits explicit commands and persists accepted normalized snapshots through the Tauri client. `SquadWorkspace` remains the semantic renderer. The only browser-storage read allowed for this feature is the narrow one-time v2–v4 migration adapter.

No dependency or architecture research spike is required. Every needed behavior can use the current React, Radix, Tauri, serde/serde_json, Vitest, Playwright and Rust test stack.

## Architecture Patterns and File Ownership
### Frontend

- Keep `MatchdayScreen.tsx` as orchestration, not the engine or repository. It currently owns the legacy preference key and must delegate table fields after migration. [VERIFIED: apps/desktop/src/matchday/MatchdayScreen.tsx]
- Keep `SquadWorkspace.tsx` as the Elenco semantic table and inspector owner; inject controlled state and command callbacks rather than replacing it. [VERIFIED: apps/desktop/src/matchday/SquadWorkspace.tsx]
- Keep labels and current football vocabulary in `matchday-ui.ts`; stable IDs are independent of those labels. [VERIFIED: apps/desktop/src/matchday/matchday-ui.ts]
- Keep football comparison behavior in `squad-sort.ts`; adapt it to ordered normalized clauses without moving football meaning into the generic engine. [VERIFIED: apps/desktop/src/matchday/squad-sort.ts]
- Recommended new ownership: one pure `table-view` module, one `squad-table-schema` module, one controller hook, one legacy migration adapter and small customizer composites.
- The engine accepts schema + state + command and returns accepted normalized state or a typed rejection. It has no React, Tauri, DOM, clock or random dependency.
- Persist stable IDs and typed values only; never persist translated labels, focus, hover, open overlays, drag state, announcements, query results or player rows.
### Rust and Tauri

- Mirror the existing injected repository/service/coordinator layering rather than placing file I/O in Tauri commands. [VERIFIED: crates/application/src/matchday.rs] [VERIFIED: crates/platform/src/matchday.rs]
- Add a table-view repository port/service at application level and its file adapter/coordinator at platform level.
- Compose a fixed sibling path under `app_data_dir`, separate from `first-playable.json`. [VERIFIED: apps/desktop/src-tauri/src/main.rs]
- Add narrow load/save/import commands beside existing `matchday_state`, `update_matchday_lineup` and `play_next_match`; preserve those commands unchanged. [VERIFIED: apps/desktop/src-tauri/src/main.rs]
- Extend `client.ts` with typed invoke wrappers; current matchday invoke wrappers are the local analog. [VERIFIED: apps/desktop/src/matchday/client.ts]
- The platform adapter validates envelope version, table/schema IDs, owner scope, bounds and sequential migration before exposing data.
- Unknown future versions, broken migration chains and corrupt values are quarantined; the application returns the valid system default plus a typed recovery outcome.
### Legacy transaction

1. Load repository state first.
2. If no completed import receipt exists, read v4, then v3, then v2.
3. Decode only supported density and column visibility/order inputs; reject unknown-only payloads.
4. Submit one import command with source version/fingerprint.
5. Await confirmed repository commit and reload/receipt.
6. Preserve non-table preferences in their existing seam, then retire only successfully imported legacy table fields/keys.
7. A failed import or save leaves legacy data untouched and shows recovery copy.

`rivallo.squad-ui.v4`, v3 and v2 plus direct get/set calls are present today. [VERIFIED: apps/desktop/src/matchday/MatchdayScreen.tsx]

## Pitfalls and Mitigations

| Risk | Required mitigation |
|---|---|
| Legacy import races repository load | Repository load precedes a single idempotent import handshake |
| Deleting v2–v4 too early | Retire only after confirmed durable receipt |
| Losing non-table v4 preferences | Split/preserve sidebar, screen, inspector and pitch values before retirement |
| NaN/infinite/huge widths | Reject non-finite values and clamp by screen schema |
| Pinned columns cover focus | Enforce count plus aggregate-width bounds and test all viewports |
| Dirty comparison changes by JSON order | Canonical normalization before semantic comparison |
| Active/default view deleted | Resolve and persist safe fallback in one accepted transaction |
| Future/corrupt payload partially applies | Quarantine whole payload and expose system default |
| Remove-then-rename gap on Windows | Prove the repository commit strategy under replacement/interruption tests; do not assume the current helper is fully atomic |
| Sort/filter loses player selection | Reconcile only by stable player ID |
| Customizer dismissal loses accepted work silently | Snapshot, explicit save/discard copy, Escape hierarchy and announcements |
| Goals view receives special code | Assert the ordinary typed filter/sort command path |
| Web/component tests lack native Tauri | Mock typed client boundary, not storage internals |
| Recovery is shown as “offline” | Use local repository wording; Phase 9 owns network states |

## Security and Data Safety

- Derive the repository path only from Tauri `app_data_dir`; never accept a filesystem path from React.
- Bound view count, label length, columns, sort clauses, filter depth/clauses, payload bytes and quarantine evidence.
- Treat invoke payloads and local JSON as untrusted; validate before mutation or persistence.
- Use fixed semantic IDs in JSON; never interpolate IDs into filenames.
- Do not log full payloads, player records or arbitrary legacy strings; log reason codes and bounded identifiers.
- Use no network, remote registry, dynamic import URL, eval or environment-secret access.
- Keep quarantine separate from the valid active repository and never overwrite it before diagnostic evidence exists.
- Confirm persistence before retiring legacy data; a save failure must be lossless and retryable.
## Open Questions (RESOLVED)

- **Windows persistence:** Plan 03 selects temporary write plus backup and deterministic reconciliation/recovery. Interruption tests must prove the repository exposes either the previous or the new valid envelope and retains quarantine evidence; the implementation must not assume blind remove-then-rename is atomic.
- **Concrete module and DTO boundaries:** Plans 01–04 select pure frontend engine/schema modules; a Rust application port, service and DTO boundary; a Rust platform file adapter and coordinator; thin Tauri load/save/import commands; typed client wrappers; and one narrow v2–v4 frontend migration adapter.
- **Dependency contract:** use only already-resolved dependencies. No new registry package/version may enter the lockfile; this is a resolved implementation contract, not a pending user decision.
- **Stable identity contract:** system/default view IDs, repository identity and related filenames are planned constants independent of localized labels; this is resolved, not a user question.
- **Preference split:** preserve the remaining non-table UI preference seam while retiring imported v2–v4 table fields only after confirmed durable persistence.
