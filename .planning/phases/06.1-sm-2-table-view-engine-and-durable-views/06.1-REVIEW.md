## Narrative Findings (AI reviewer)

### CR-01: A legacy-imported view cannot be deleted

**Resolution:** Resolvido. A exclusão preserva o receipt como tombstone durável, o decoder aceita receipts cuja visualização importada já foi excluída e as regressões cobrem importação, exclusão e recarga sem reimportação.

**Files:** `apps/desktop/src/matchday/use-squad-table-view.ts:1055-1062`; `crates/application/src/table_view.rs:893-913`; `crates/application/src/table_view.rs:1279-1287`; `apps/desktop/src/matchday/client.ts:499-506`

**Issue:** The frontend delete candidate removes every receipt whose `importedViewId` matches the deleted view. Ordinary Rust saves require `legacy_import_receipts` to remain byte-for-byte equal, so that candidate is rejected as `table_view.invalid_transition`. Rust's own lifecycle delete deliberately leaves receipts in place, confirming that they are durable import tombstones. However, preserving the receipt in the frontend is not currently sufficient because the TypeScript decoder rejects a receipt whose imported view is no longer present.

**Impact:** A normal mutable, user-owned view created by the v2-v4 legacy importer cannot complete the advertised delete lifecycle. The UI reaches persistence and then reports a generic save failure. The incompatible decoder also cannot represent the state produced by the application-owned delete policy.

**Fix:** Define legacy receipts explicitly as durable tombstones independent of current view membership. Preserve them when deleting a view, remove the TypeScript extant-view check, and keep the Rust ordinary-transition equality rule. Add a decoder fixture plus controller/Rust/browser regressions that import, delete, reload, and verify the same legacy payload is not imported again.

### CR-02: Unversioned save and retry continuations can replace newer proposal edits

**Resolution:** Resolvido. Propostas, operações e continuações agora possuem versionamento monotônico; retries são reconstruídos a partir da proposta atual e respostas antigas não podem sobrescrever uma edição mais nova.

**Files:** `apps/desktop/src/matchday/use-squad-table-view.ts:417-421`; `apps/desktop/src/matchday/use-squad-table-view.ts:667-715`; `apps/desktop/src/matchday/use-squad-table-view.ts:795-799`; `apps/desktop/src/matchday/use-squad-table-view.ts:860-901`; `apps/desktop/src/matchday/TableViewCustomizer.tsx:434-450`; `apps/desktop/src/matchday/TableViewCustomizer.tsx:481-669`

**Issue:** A failed mutation stores only the old repository candidate and preview, without the proposal version that produced them. `retry()` resubmits that snapshot, and `persistCandidate()` first calls `updateProposal(preview)`. Because column, visibility, pinning, resize, and search interactions remain active while persistence is pending, edits made after a failure are silently replaced by the stale preview on retry. A second variant occurs during dirty activation: the first save correctly preserves a proposal changed while awaiting confirmation, but `activate()` then proceeds without rechecking the proposal version and applies the target view preview, discarding those newer edits.

**Impact:** Users can lose table changes even though the controller intentionally preserves new in-memory work from the first save response. The durable repository, baseline, and visible proposal can also describe different user actions until another reload or save.

**Fix:** Attach a monotonic proposal version and operation token to every submitted and failed candidate. Retry must rebuild from the current proposal or require explicit confirmation before restoring an older snapshot. After every awaited save, recheck the expected version before activation; if it changed, keep the proposal dirty and stop the continuation. Add deferred-promise tests for edit-after-failure/retry and edit-during-save/save-and-open.

### CR-03: Dismissing a busy transition does not cancel its navigation or deletion

**Resolution:** Resolvido. O cancelamento invalida apenas a continuação pendente; a persistência já enviada pode terminar sem ativar, navegar ou excluir depois de Escape ou “Continuar nesta visualização”.

**Files:** `apps/desktop/src/matchday/SavedViewSelector.tsx:381-402`; `apps/desktop/src/matchday/SavedViewSelector.tsx:525-555`; `apps/desktop/src/matchday/MatchdayScreen.tsx:511-520`; `apps/desktop/src/matchday/MatchdayScreen.tsx:535-565`; `apps/desktop/src/matchday/MatchdayScreen.tsx:712-745`

**Issue:** The dirty dialog keeps Escape and “Continuar nesta visualização” active while a save is pending, and the name dialog likewise keeps its close/dismiss path active. Dismissal clears the rendered dialog state, but the async handler has already captured the original transition target. When persistence confirms, it unconditionally calls the captured continuation. For a view it activates that view, for a screen it navigates, and for a dirty delete it proceeds to delete. Save-as completion has the same captured-continuation behavior.

**Impact:** The UI tells the user they are remaining on the current view, yet a later response can move them elsewhere or delete a view. This is a destructive violation of the user's explicit cancellation decision.

**Fix:** Give each continuation an identity stored in a ref/state machine. Dismissal should invalidate only the post-save continuation, not the already-submitted persistence. After the await, verify that the continuation is still current before activating, navigating, or deleting. Add deferred-save regressions for Escape/Continue across view, screen, delete, and immutable save-as transitions.

## Warnings

### WR-01: The v1 migration turns an emptied filter root into invalid `null`

**Resolution:** Resolvido. Migrações aceitam `filter: null` e filtros compostos apenas por colunas removidas, normalizando-os para uma raiz vazia estável e recarregável.

**Files:** `crates/platform/src/table_view.rs:862-867`; `crates/platform/src/table_view.rs:1361-1373`; `crates/application/src/table_view.rs:1450-1454`; `apps/desktop/src/matchday/client.ts:393`

**Issue:** `migrate_storage_v1_to_v2` removes obsolete `removedColumn` filter clauses, but writes `filter = null` when that removal empties the root and also leaves an already-null historical value unchanged. Current application validation requires `Some(TableFilterGroup)`, and the TypeScript decoder also expects a concrete root group. The post-migration candidate is therefore classified as `InvalidPayload`.

**Impact:** A recoverable v1 repository whose only filter referenced the removed column is quarantined wholesale instead of retaining its saved views with an empty filter. Existing migration tests remove the obsolete column definition but do not exercise null, removed-only, or mixed nested filters.

**Fix:** Retain the existing root with empty `children`, or synthesize the application-owned stable empty root when the legacy value is null. Add v1 fixtures for null-root, removed-only, and mixed nested filters and assert a `Migrated` result followed by a clean reload.

### WR-02: Quarantine diagnostic writes gate recovery of otherwise valid data

**Resolution:** Resolvido. A gravação diagnóstica de quarantine passou a ser best effort e não oculta uma geração válida quando payload ou evidência não podem ser persistidos.

**Files:** `crates/platform/src/table_view.rs:261-305`; `crates/platform/src/table_view.rs:345-386`; `crates/platform/src/table_view.rs:410-432`; `crates/application/src/table_view.rs:754-758`

**Issue:** Reconciliation persists quarantine payload/evidence before returning a valid active repository with stale transaction artifacts and before promoting a valid temp/backup when active is absent. Any auxiliary diagnostic-write failure becomes `Unavailable`, which the application maps to the system-default fallback.

**Impact:** Readable, valid saved views can remain hidden indefinitely merely because `.quarantine.payload` or `.quarantine.json` is unwritable, full, or occupied by a directory. The interruption suite only exercises writable directories and requires evidence success.

**Fix:** Separate successful recovery/promotion from best-effort diagnostic persistence. Return the valid generation and surface an explicit evidence-write warning; retain invalid artifacts in place when they cannot be safely quarantined. Add deterministic diagnostic-write-failure tests for valid-active and valid-temp/backup recovery.

### WR-03: The 1024px customizer violates its width and persistent-header contract

**Resolution:** Resolvido. O customizador usa largura de `24rem`, respeita o viewport, mantém cabeçalho e fechar persistentes, limita-se pela altura disponível, usa o corpo como único scroller e preserva as ações sticky.

**Files:** `apps/desktop/src/matchday/matchday.css:3180-3183`; `apps/desktop/src/matchday/matchday.css:3710-3729`; `apps/desktop/src/ui/primitives/disclosure.tsx:141-149`; `browser-tests/matchday.spec.ts:1356-1464`

**Issue:** The desktop customizer is capped at `34rem` (544px), and the compact fixed wrapper is capped at `32rem` (512px), despite the approved contract calling for roughly 384px desktop and no more than 420px at 1024px. In the tracked 1024x768 baseline, the sheet covers about half the viewport and its title/close row is not visible, leaving the content without orientation or an obvious pointer dismissal.

**Impact:** The overlay hides the table columns users are trying to preview and weakens both immediate verification and discoverable dismissal at the minimum supported viewport.

**Fix:** Constrain the anchored surface to about `24rem` and the compact sheet to at most 420px. Make the popover a two-row layout with a persistent header/close row and scroll only the content body. Extend the 1024 browser assertion to require the heading and close control to be visible and the measured width to satisfy the contract.

### WR-04: The WCAG inventory overstates the states and zoom behavior actually proven

**Resolution:** Resolvido. A matriz tipada exige cobertura exata entre requisitos e amostras aplicáveis, distingue corretamente controles inativos de alegações normativas AA e o cenário de zoom de 200% mede clipping, colisões e preservação do nome longo acessível.

**Files:** `browser-tests/matchday.spec.ts:1207-1293`; `browser-tests/matchday.spec.ts:1301-1354`; `tooling-tests/phase-6-1-table-view-boundaries.test.mjs:817-853`

**Issue:** The browser test declares fifteen operational states and only asserts the inventory length, while the computed samples cover a subset of those states and controls. The boundary test checks that inventory/sample-related strings exist, not that every declared state maps to an applicable computed sample. The 200% scenario verifies reflow, overall overflow, reachability, and action visibility, but performs no per-label clipping or overlap measurement.

**Impact:** Active controls, truly disabled boundaries, resize handles, live header controls, provenance/default labels, destructive actions, and long Portuguese labels can regress while the suite still emits a green “15-state” WCAG artifact. The phase therefore does not yet prove its own all-state WCAG 2.2 AA and no-clipping/no-overlap acceptance claim.

**Fix:** Represent required state/control samples as a typed matrix and assert exact emitted-sample coverage. Add computed ratios for every applicable entry. At 200% zoom, add explicit text/container geometry checks for clipping and collisions, or a stable zoomed visual assertion plus targeted DOM measurements.
