## Scope and Preservation Boundary

This phase incrementally upgrades the existing **Elenco** table. It does not redesign the application shell or create a replacement squad screen.

Preserve without visual or behavioral regression:

- the AppShell, top bar, sidebar, routes and full-screen desktop behavior;
- the Elenco page hierarchy, real 18-player fixture, shirt numbers, minifaces, nationality fallbacks, GOL-to-ATA ordering, XI controls and player inspector;
- the separate TĂĄticas screen and all current matchday/gameplay behavior;
- semantic native `<table>` relationships, stable player-ID selection and existing focus behavior;
- Rivallo tokens, primitives, Radix behavior packages and preference migration behavior outside table-view ownership.

Do not introduce grouping controls, fake data, new ratings, a second product table, SQLite, selectable career identity, server-query controls, cache/offline/sync claims, shared-view transport or unrelated shell changes.

The owning contract is fixed:

| Property | Value |
|---|---|
| `tableId` | `squad.primary` |
| `schemaVersion` | `1` |
| `ownerScope` | `local-fixed` |
| Active data mode | Bounded `client-pagination`, page size 25; all 18 fixture rows fit in one window |
| Grouping | Unsupported; no grouping control or empty placeholder renders |
| Persistence authority | Rust application/platform repository behind Tauri commands |
| React authority | Controlled rendering, transient interaction and command emission only |

`Mostrar somente gols` is a normal user-owned view using the common sort/filter pipeline. It must never produce a `goalsOnly` flag or a special renderer/query branch.

## Color

The phase explicitly follows a restrained **60/30/10** distribution:

| Role | Token/value | Usage |
|---|---|---|
| Dominant (60%) | `--rv-color-canvas` / `oklch(0.145 0.014 190)` | workspace and scroll-area ground |
| Secondary (30%) | `--rv-color-surface` / `oklch(0.185 0.018 190)` | table panel, inspector and customizer; `surface-raised` is its tonal control/overlay layer |
| Accent (at most 10%) | `--rv-color-action-primary` / `oklch(0.72 0.15 155)` | one primary commit action and confirmed-positive save/default feedback |
| Destructive | `--rv-color-danger` / `oklch(0.68 0.18 25)` | delete-view action, invalid state and persistence failure only |

Accent is reserved for:

1. the enabled primary `Salvar visualizaĂ§ĂŁo` action;
2. concise confirmed-success feedback after save or set-default;
3. no more than one affirmative focal action inside an open lifecycle dialog.

It is not used on every interactive control, every icon, inactive tabs, borders, table headers or decorative surfaces.

Semantic colors are not additional decorative accents:

- `color-info` and `color-focus` identify keyboard focus, the active view, selected row and informational status;
- `color-selection` supplies the selected/current surface together with text or `aria-current`;
- `color-warning` accompanies the textual `AlteraĂ§Ăľes nĂŁo salvas` or recovery message;
- `color-danger` always appears with an icon and explicit error/destructive copy;
- `color-premium` and club colors gain no new use in this phase.

No meaning depends on color alone. Do not add gradients, glass, glow, wide decorative shadow or colored side stripes.

## Visual Hierarchy

### Primary focal point

The squad table remains the visual anchor. Player identity, position and row scan paths have higher visual weight than customization controls. The active saved-view name and textual dirty state sit in the table-panel header, immediately before `Configurar tabela`.

### Secondary and tertiary levels

1. The existing screen title and fixture context establish location.
2. The saved-view selector, filter/sort toolbar and table controls establish the current lens.
3. The player inspector remains contextual and visually subordinate to the table.
4. The portalled customizer is elevated only while open; it must not resemble a new page or a card inside another card.

Use tonal separation, one restrained border and existing elevation tokens. Preserve player faces, real shirt numbers and XI state as semantic content, not decoration.

## Saved-View Lifecycle

The table-panel header contains, in order:

1. selector labelled `VisualizaĂ§ĂŁo da tabela`;
2. provenance text (`PadrĂŁo do sistema`, `Minha visualizaĂ§ĂŁo` or `Somente leitura`);
3. textual dirty state `AlteraĂ§Ăľes nĂŁo salvas` when applicable;
4. `Configurar tabela`;
5. contextual view menu.

The lifecycle supports activate, create, duplicate, rename, delete, set-default, reset, save and the save/discard/continue decision before a dirty view is replaced.

| Provenance | Visible behavior |
|---|---|
| `system-default` | Activate, duplicate and define as user default. Rename/delete are absent. Editing starts a dirty proposal and saving asks for a new user-owned name. |
| `user-owned` | Activate, edit, save, duplicate, rename, delete, reset and define as default. |
| `shared-read-only` | Activate and duplicate. It shows `Somente leitura`; direct edit/rename/delete actions are absent. This is provenance support only, not a network-sharing claim. |

The selector shows the system view first, user views alphabetically and read-only views last. Default and active states use text/icon plus semantics; they never depend only on color.

### Dirty transitions

Switching view, deleting the active view or leaving Elenco while dirty opens one confirmation with:

- `Salvar e abrir â{nome}â`;
- `Descartar e abrir â{nome}â`;
- `Continuar nesta visualizaĂ§ĂŁo`.

Failed persistence keeps the proposal dirty and visible. A successful save updates the baseline, clears the dirty text and returns focus to the invoking control.

### Ordinary configured-view proof

A user creates `Mostrar somente gols` through the normal create/edit/save flow:

- required columns plus `goals` are visible;
- `goals` sorts descending, with player name as deterministic tie-breaker;
- a typed `goals > 0` clause is enabled;
- grouping remains empty;
- the resulting view can be activated, renamed, duplicated, set default, reset and deleted like any other user-owned view.

## Customizer Interaction Contract

`Configurar tabela` opens one controlled, portalled, non-modal surface anchored to the existing table controls. It becomes sheet-like only at 1024px. It does not replace, obscure entirely or nest another card around the table.

The surface contains:

1. heading `Personalizar tabela`;
2. search field `Buscar colunas`;
3. ordered column rows with visibility, required explanation, move control, finite width control and pin menu;
4. density choices `Compacta`, `PadrĂŁo`, `ConfortĂĄvel`;
5. `Restaurar colunas` for visibility/order/width/pinning only;
6. `Restaurar visualizaĂ§ĂŁo` for the complete active baseline;
7. footer actions `Descartar ajustes` and `Salvar visualizaĂ§ĂŁo`.

Changes preview immediately in the table but remain a reversible proposal until saved. `Descartar ajustes`, outside dismissal or Escape with no active sub-operation restores the snapshot from when the surface opened. If a move/resize operation is active, the first Escape cancels only that operation; a later Escape dismisses the surface.

Required visibility controls remain keyboard-reachable with `aria-disabled="true"` and expose their inline explanation. Unsupported grouping never renders as disabled copy, a placeholder or an âEm breveâ option.

## Sort, Filter and Density Contract

- Header activation cycles ascending â descending â removed while retaining the current visible direction and `aria-sort`.
- Shift+activation adds or updates a secondary sort; Elenco supports at most three ordered clauses.
- The sort editor names each clause, direction and priority. Removing a clause moves focus to the next clause, or the add-sort control when none remain.
- `Todos / Titulares / Reservas`, `Setor`, `Status` and optional `PosiĂ§ĂŁo` are normalized into the shared typed filter tree and are durable in saved views.
- `Prontos` normalizes to `matchFitness >= 90 AND condition >= 90`; `AtenĂ§ĂŁo` normalizes to `matchFitness < 90 OR condition < 90`.
- The existing global player-name query remains explicitly transient: it filters the current window but does not dirty or persist the view.
- Filter depth is at most two groups and twelve clauses. Invalid values are rejected with an exact textual reason; no silent coercion occurs.
- `Limpar filtros do elenco` clears durable table filters through engine commands but does not alter columns, density or sorting.
- Density changes preserve the existing 44/48/54px row geometry and become part of the saved view.

Sorting, filtering, column ordering and pinning preserve focused/selected players by stable player ID, never row index.

## Accessibility and Motion

- Target WCAG 2.2 AA: 4.5:1 normal text, 3:1 non-text controls/focus, visible focus at every step.
- Preserve native table headings, row headers and `aria-sort`; customizer controls reference the localized column label but store only stable IDs.
- Portalled surfaces receive an accessible heading and description. Non-modal customizer focus is not trapped; destructive/dirty confirmations use the existing controlled dialog focus pattern.
- Required, hidden, pinned, sorted, filtered, active, default, dirty, recovered and failed states use text/semantics in addition to color or icon.
- Tooltips only supplement a stable accessible name.
- Motion uses existing 100ms feedback, 150ms control and 200â250ms state transitions, ease-out, opacity/transform only. No layout-property animation, bounce or entrance choreography.
- `prefers-reduced-motion: reduce` makes transitions immediate without removing feedback, announcements or focus restoration.
- Verify at 200% zoom and with long Portuguese view names. Labels may truncate visually only when the full name remains programmatically and tooltip-accessible.
